# Session Summary: Source Resolution & Package Index

**Date**: 2026-03-02
**Status**: Research complete. Decisions 32-58 captured. Linear needs
alignment (no premature changes). Ready for spec work once Linear is
reviewed.

## The Core Insight

The install command can't work without a source resolution layer, and
the original "scope = GitHub owner" convention breaks for the dominant
real-world case: monorepo skill catalogs (one repo, many packages).
350K+ skills already exist on GitHub. The tool that makes them
installable with zero upstream cooperation wins.

## What We Designed

**Per-package metadata index** served as static files over HTTP. One
JSON file per package at `/@scope/name.json`. Scales to any size.
Resolution is a single HTTP GET.

**Two install strategies** (Composer's dist/source model):

- **Dist** (native packages): author published a `.aipkg` archive.
  CLI downloads it, verifies SHA-256 from the metadata, unpacks.
- **Source** (virtual packages): no archive exists. CLI fetches the
  source directory from GitHub at a pinned ref, packs locally, installs.

**Helm-style bundled dependencies**. Dependencies resolved at pack time,
bundled inside the parent archive. Install extracts everything from one
archive. No transitive resolution at install time. Author controls
dependency visibility (default: internal, not exposed to the user).

**Central index** hosted at `packages.aipkg.dev` (`ai-interop/packages`
repo, GitHub Pages). Two entry types: `"package"` entries for published
`.aipkg` archives (versioned, with dist URLs + SHA-256) and `"recipe"`
entries for virtual packages (unversioned, pointing at upstream source).
Recipes can be written by hand or generated by automation (a crawler is
one possible automation, not on the critical path).

## Key Decisions (32-58)

| # | Decision | One-liner |
|---|----------|-----------|
| 32 | Index-based discovery | No GitHub API on the install path |
| 33 | Central index by aipkg project | Homebrew model for cold start |
| 34 | No @virtual namespace | Virtual packages use upstream's scope |
| 35 | Source-resolved virtuals | CLI fetches + packs on-the-fly, no pre-building |
| 36 | SHA-256 in the index | No sidecar fetch during install |
| 37 | Resolution priority | Project repos first, central index last |
| 38 | No version ranges | Latest or exact match only |
| 39 | Single source type | All sources serve same per-package format |
| 40 | Explicit version derivation | Maintainer assigns versions, not automated |
| 41 | Per-package files | Not a single index.json (won't scale to 350K) |
| 42 | Dist vs. source strategies | Determined by metadata entry, transparent to user |
| 43 | Automated crawler | Scans GitHub for SKILL.md, generates metadata only |
| 44 | Helm-style bundled deps | Deps resolved at pack time, bundled in archive |
| 45 | Virtual = single artifact | One skill/prompt/command per virtual package |
| 46 | Scope ≠ repository | Scopes are identifiers, not repo pointers |
| 47 | GitHub Releases = recommendation | Hosting-agnostic; dist URL can point anywhere |
| 48 | Central index at aipkg.dev | Owned domain, CNAME to static hosting under ai-interop |
| 49 | Dep visibility default internal | Author controls; bundled deps hidden from user by default |
| 50 | Repositories field deferred | Needs fresh design based on index model |
| 51 | Scope claiming process | GitHub org verification, manual for v1 |
| 52 | Unified index: package + recipe schemas | Two entry types, one index, clean separation |
| 53 | Recipe entries unversioned | No redundant version lists; consumer-side pinning |
| 54 | Crawler is optional automation | Not on critical path; hand-written recipes work |
| 55 | dist wrapper kept for extensibility | Future methods (OCI etc.) can be added as siblings |
| 56 | HTTP-only transport | No file:// scheme; static file servers for local testing |
| 57 | Env var for registry override | Defer repositories config; AIPKG_REGISTRY for testing |
| 58 | Feature split by entry type | Package install first, recipe install second |

## The Four Package Flavors

```text
                    Dedicated repo          Monorepo / catalog
                 ┌─────────────────────┬─────────────────────────┐
  Has aipkg.json │  A. Native 1:1      │  B. Native catalog      │
  (packaged)     │  dist strategy      │  dist strategy          │
                 ├─────────────────────┼─────────────────────────┤
  No aipkg.json  │  C. Virtual 1:1     │  D. Virtual catalog     │
  (unpackaged)   │  source strategy    │  source strategy (350K+)│
                 └─────────────────────┴─────────────────────────┘
```

All four resolve through the same index. The CLI checks the `type` field:
`"package"` entries use the dist strategy (download archive), `"recipe"`
entries use the source strategy (fetch from GitHub, pack locally).

## Open Questions (not blocking Feature A)

1. **GitHub API for source installs** (Feature B): tarball endpoint vs.
   contents API vs. sparse checkout
2. **Rate limits** (Feature B): 60 req/hr unauthenticated. GitHub token
   support needed.
3. **Search/discovery**: separate from resolution. Integrate with
   skillsmp.com?
4. **Consuming skillsmp.com data**: build own crawler, consume their
   data, or partner?
5. **Non-skill artifact types**: how does the crawler discover prompts,
   commands, agents?

## What's NOT Settled Yet

- Linear backlog needs alignment with this architecture. Proposed changes
  need review and explicit approval before execution.
- The index metadata schemas (package + recipe) are well-shaped but need
  spec-level rigor (JSON Schema, validation rules, edge cases).
- Constitution Principle IV still references `@virtual/owner:repo` and
  "GitHub-first distribution via GitHub Releases." Wording needs amendment
  to match the index-based model.
- The `require` field exists in `spec/schema/package.json` but does nothing
  yet (Principle I concern). Needs reworking as part of the install spec.
- The relationship between AIPKG-5 and AIPKG-52 (merge or keep separate?).
- Whether AIPKG-9, AIPKG-13, AIPKG-15 should be closed or kept as re-scoped items.
- GitHub API strategy for source installs (tarball vs contents API).
- Rate limit handling (60 req/hr unauthenticated).
