# Source Resolution Architecture: Session Briefing

**Date**: 2026-03-02
**Session**: Brainstorm on source resolution, package index, and install
architecture. Triggered by attempting to spec AIPKG-10 (Install Command)
and discovering the resolver architecture was an unresolved prerequisite.

**Pre-requisite reading**: Familiarity with aipkg's manifest schema
(`aipkg.json`), the `.aipkg` archive format (002-archive-format-pack),
and the constitution (`.specify/memory/constitution.md`).

---

## The Problem

The CLI needs to map a package name (`@scope/name`) to installable
content. The original design assumed "scope = GitHub owner, name = repo
name" and packages are fetched from GitHub Releases.

This breaks for the dominant real-world case: **monorepo skill catalogs**.
Repos like `ComposioHQ/awesome-claude-skills` contain 30+ individually
installable skills under one repo. The scope/name convention can't map
`@composiohq/code-review` to the right subdirectory inside that repo.

350K+ skills already exist on GitHub (per skillsmp.com). Making them
installable with zero upstream cooperation is the strategic opportunity.

## Architecture: What We Designed

### Per-package metadata index

A directory of static JSON files served over HTTP. One file per package
at `/@scope/name.json`. Resolution is a single HTTP GET.

Hosted at `packages.aipkg.dev` (owned domain, CNAME'd to GitHub Pages
under the `ai-interop` org). The aipkg project maintains a central
index (Homebrew model for cold start). Projects can configure additional
sources that take priority over the central index.

### Two index entry types, two install strategies

The index serves two JSON schemas, discriminated by a `type` field:

- **`"package"`** (dist strategy): author published `.aipkg` archives.
  Entry has a `versions` block; each version has a `dist` with URL +
  SHA-256. CLI downloads, verifies, unpacks.
- **`"recipe"`** (source strategy): no archive exists. Entry has an
  `artifact` type and `source` pointer (repo + path). No versions block.
  CLI fetches from GitHub, packs locally, installs.

The user doesn't choose. The `type` field determines the strategy.
Recipe entries carry no version history; versions are tracked on the
consumer side (pinned ref in `require`).

### Helm-style bundled dependencies

Dependencies are resolved at **pack time** and bundled inside the parent
archive as nested `.aipkg` files in a `deps/` directory. Install
extracts everything from one archive. No transitive dependency resolution
at install time.

Author controls dependency visibility: `internal` (default, not exposed
to the user's IDE) or `public` (exposed as if installed directly).
This matters because AI skills that delegate to other skills are
implementation details, not things the user wants cluttering their tool.

### Virtual packages (recipes)

Defined by recipe entries in the index. Each recipe points at upstream
content (repo + path) and declares the artifact type. Use the upstream
author's scope (e.g., `@composiohq/code-review`, not
`@virtual/composiohq:code-review`). Each virtual package is a single
artifact. Graduation from virtual to native is seamless (same name,
better install performance).

Recipes can be written by hand or generated by automation (a crawler
is one possible automation, but not on the critical path).

## Decisions

This session produced decisions 32-58. Full rationale in
[02-design-decisions.md](02-design-decisions.md). Highlights:

| # | Decision | Why it matters |
|---|----------|----------------|
| 32 | Index-based discovery | No GitHub API on install path |
| 34 | No @virtual namespace | Seamless graduation, simpler UX |
| 35 | Source-resolved virtuals | No pre-building, no DefinitelyTyped trap |
| 41 | Per-package files | Scales to 350K+ (single index.json doesn't) |
| 44 | Helm-style bundled deps | Self-contained archives, zero install-time resolution |
| 48 | Central index at aipkg.dev | Owned domain, decoupled from GitHub handle |
| 52 | Unified index: package + recipe schemas | Two entry types, one index, clean separation |
| 53 | Recipe entries unversioned | No redundant version lists; consumer-side pinning |
| 54 | Crawler is optional automation | Not on critical path; hand-written recipes work |
| 56 | HTTP-only transport | No file:// scheme; static file servers for local testing |
| 57 | Env var for registry override | Defer repositories config; `AIPKG_REGISTRY` for testing |
| 58 | Feature split by entry type | Package install first, recipe install second |

## Next Features to Spec

Split into two features by entry type (Decision 58), not by transport.

### Feature A: Package Install (dist strategy)

The smallest useful thing: install `type: "package"` entries from an
index. Closes the pack-to-install loop.

**In scope:**

- Per-package metadata schema (`type: "package"` entries with versioned
  `dist` blocks)
- Resolution algorithm (GET `$AIPKG_REGISTRY/@scope/name.json`, find
  version, download archive)
- Download archive from `dist.url`, verify `dist.sha256`
- Unpack to `.aipkg/packages/`
- Extract bundled deps from `deps/` inside archive
- Update `require` in project manifest (exact version pin)
- `aipkg install` (no args) to reinstall from require
- `AIPKG_REGISTRY` env var for registry override (default:
  `packages.aipkg.dev`)
- Reference documentation

**Out of scope (deferred to Feature B or later):**

- Recipe/source strategy (Feature B)
- Full `repositories` manifest config (deferred; env var is enough)
- Direct URL install (`aipkg install https://...`)
- Dependency bundling in pack (AIPKG-47b)
- Crawler and central index population (AIPKG-53)
- Publish command (AIPKG-16)
- Search/discovery, version ranges, adapters

**Testing approach:** Hand-craft a directory with per-package JSON files
and `.aipkg` archives. Serve with `python -m http.server`. Point
`AIPKG_REGISTRY` at `http://localhost:8000`. Existing archives from
`aipkg pack` provide real test data.

### Feature B: Recipe Install (source strategy)

Adds `type: "recipe"` support. Unlocks the 350K+ existing skills.

**In scope:**

- Recipe entry schema (`type: "recipe"` with artifact type + source
  pointer)
- Source strategy (fetch from GitHub at ref, pack locally, install)
- Ref resolution (HEAD of default branch, or pinned ref from require)
- GitHub API integration (tarball or contents API)
- Rate limit handling, optional GitHub token support
- Direct URL install (`aipkg install https://...`) as convenience

**Depends on:** Feature A (same unpack/require/resolution infrastructure).

## Constitution Impact

Principle IV (Cold Start First) needs amendment. Two things changed:

1. `@virtual/owner:repo` namespace dropped (Decision 34). Virtual
   packages use the upstream's scope.
2. "GitHub-first distribution via GitHub Releases" replaced by
   index-based discovery with per-package metadata. GitHub Releases
   is now a hosting *recommendation*, not an architectural requirement.

The spirit of Principle IV is preserved (zero-config install, cold start
focus). The wording needs to match the new architecture.

Note: the `require` field was subsequently removed from `spec/schema/aipkg.json`
per Principle I ("fully implemented or fully absent"). It will be reintroduced
when pack-time dependency bundling is specced.

## Open Questions (not blocking spec work)

1. GitHub API strategy for source installs (tarball vs contents API)
2. Rate limit handling (60 req/hr unauthenticated)
3. Consuming skillsmp.com data for the crawler
4. Non-skill artifact discovery in virtual packages
5. Pre-built CDN for popular virtual packages (future optimization)
